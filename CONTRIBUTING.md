# Contributing to BSP TypeScript SDK

Thanks for your interest in contributing to the Biological Sovereignty Protocol TypeScript SDK. This guide walks through local setup, testing, and the pull request flow.

## Prerequisites

- Node.js `>=18.0.0`
- npm `>=9` (bundled with Node 18+)
- Git

## Local setup

```bash
git clone https://github.com/Biological-Sovereignty-Protocol/bsp-sdk-typescript.git
cd bsp-sdk-typescript
npm install
```

## Build

```bash
npm run build      # compiles TypeScript to dist/
npm run typecheck  # type-check only (no emit)
```

## Run tests

```bash
npm test            # single run
npm run test:watch  # watch mode during development
```

Tests live under `tests/` and use `jest` + `ts-jest`.

## Lint

```bash
npm run lint
```

The project uses ESLint with `@typescript-eslint`. Please fix lint errors before opening a PR.

## Generate documentation

API documentation is generated by [TypeDoc](https://typedoc.org/):

```bash
npm run docs
```

Output is written to `docs/`. The folder is git-ignored — regenerate locally before reviewing.

## Run examples

Runnable examples live in `examples/`:

```bash
npx ts-node examples/01-create-beo.ts
npx ts-node examples/02-grant-consent.ts
npx ts-node examples/03-submit-biorecord.ts
npx ts-node examples/04-destroy-beo.ts
```

Without `BSP_RELAYER_URL` set, examples run in simulated mode (no on-chain writes).

## Code style

- Follow the existing formatting (2-space indent, single quotes, no semicolons where already absent).
- Public APIs must be typed — no `any` unless explicitly justified in a comment.
- Prefer immutable patterns; avoid mutation of inputs.
- Names use `camelCase` for values, `PascalCase` for types/classes.
- Add JSDoc blocks on all exported symbols so TypeDoc picks them up.

## Commit messages

Use [Conventional Commits](https://www.conventionalcommits.org/):

```
feat(access): add batch revoke API
fix(crypto): guard hexToBytes against odd-length input
docs(sdk-ts): expand BEOClient usage examples
```

## Pull requests

1. Fork the repo (or branch if you have write access).
2. Create a feature branch: `git checkout -b feat/my-thing` or `fix/bug-xyz`.
3. Write tests. Every new public function or behavior needs coverage.
4. Run `npm test && npm run lint && npm run typecheck` locally — CI will enforce this.
5. Update `CHANGELOG.md` under the `[Unreleased]` section.
6. Open a PR against `main` with a clear description and link to any related issue.
7. A maintainer will review. Please be patient and responsive to feedback.

## Security

Do **not** file public issues for security vulnerabilities. See `SECURITY.md` for private disclosure instructions.
